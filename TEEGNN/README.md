# TEE-GNN OP-TEE Skeleton

This directory is a migration skeleton for a real OP-TEE deployment. It is not expected to build on this machine unless the OP-TEE client headers, libraries, and TA dev kit are installed.

## Structure

- `host/main.c`: REE-side host app. It opens a TEEC session and invokes the TEE-GNN command surface.
- `ta/teegnn_ta.c`: Trusted Application entry points and command dispatch.
- `ta/include/teegnn_ta.h`: UUID and command IDs shared by host and TA.
- `ta/sub.mk`: TA source list.
- `ta/user_ta_header_defines.h`: OP-TEE TA metadata.
- `ta/manifest.json`: lightweight migration notes for packaging.

## Commands

- `TEEGNN_CMD_INIT_CONTEXT`
- `TEEGNN_CMD_RESTORE_AGGREGATION`
- `TEEGNN_CMD_APPLY_ACTIVATION`
- `TEEGNN_CMD_FINALIZE_RESULT`

The host currently passes placeholder buffers. In the full port, the host should load masked tensors generated by the REE side and pass shared memory references to the TA. The TA should restore aggregation, apply low-rank correction, run activation/final argmax, and return verification metadata.

## Randomness

The existing repository CSPRNG lives in `../CSPRNG`. A normal Linux process can link it directly, as the `nonTEE` simulator does. A real TA may not be able to link that code unchanged because OP-TEE has a restricted libc and its own crypto/runtime APIs. The TA therefore exposes `teegnn_secure_random()` as an adapter boundary with a TODO to either port `CSPRNG/csprng.c` into the TA build or replace it with an OP-TEE backed secure implementation.

## Example Build Shape

```bash
cd TEEGNN
make OPTEE_CLIENT_EXPORT=/path/to/optee_client/out/export/usr \
     TA_DEV_KIT_DIR=/path/to/optee_os/out/arm/export-ta_arm32
```
